🩺 What is QueryDoctor?

In plain English

QueryDoctor reads a database query (written in SQL, the language used to ask a database for data) and tells you, in plain language, whether it's actually correct — and if not, exactly what's wrong and how to fix it. Think of it like a spell-checker or a proofreader, but for database questions instead of sentences. You paste your query in, and within a second it hands back a report card: a health score out of 100, a list of problems explained the way you'd explain them to a colleague (not in error-code jargon), and a cleaned-up version you can copy and use right away.

You don't need to know how databases work internally, and nothing you paste in is ever saved, shared, or looked at by anyone — it's checked, shown to you, and then it's gone.

Step by step, what happens when you use it

  1. You paste a query and pick which "dialect" it's written in (different databases — BigQuery, Postgres, MySQL, and 7 others — have slightly different SQL vocabulary, the same way British and American English spell a few words differently).
  2. You tap "Diagnose my SQL." Nothing is sent anywhere except that one check — no account, no login, no saved history on any server.
  3. First, it checks the query actually makes sense at all — like checking a sentence is grammatically valid before checking if it's well-written. If it doesn't, you get the exact spot the problem is and a plain-English guess at what you meant.
  4. If it's valid, it's screened against 34 known mistake patterns — real, common ways queries quietly do the wrong thing (like accidentally multiplying your results, or a comparison that can never be true) even though they "look" fine and would never error out on their own. Each one found is explained in a sentence or two, not a code.
  5. You get a health score (0–100) summarizing all of that at a glance, plus (when relevant) a cleaned-up version, a rewritten/optimized version, some fixes applied automatically with one click, and a translation into a different database's dialect.

For developers: what's actually under the hood

  • Zero LLM, fully deterministic. No AI model is involved and nothing is "guessed" — the engine is sqlglot, a pure-Python SQL parser, used to build a real abstract syntax tree (AST) and reason about it structurally. Same input always produces the exact same output, with no hallucination risk and no per-request inference cost.
  • 34 hand-written AST rules (plus a token-level missing-comma detector) catch real correctness bugs, not just style nits — silent NULL-comparison traps, outer joins quietly turned into inner joins by a stray WHERE clause, aggregates mixed with ungrouped columns, self-joins missing a distinguishing alias, a table alias accidentally leaking into a column's dataset qualifier, and more. See the full rule list.
  • The optimizer suggestion doesn't just trust itself. sqlglot's own rewrite engine can, in certain correlated-subquery shapes, silently change a query's actual results — QueryDoctor detects that risk structurally and independently re-verifies the rewrite by executing both the original and optimized query against synthetic in-memory SQLite data and comparing results, before ever showing you a "cleaner" version.
  • Optional schema awareness. Paste real CREATE TABLE DDL or a JSON schema to unlock Unknown table/Unknown column checks — entirely opt-in, parsed locally, never required.
  • -- noqa suppression (matching sqlfluff's own convention) to dismiss a finding you've already reviewed, and a small set of zero-ambiguity auto-fixes (x = NULL → IS NULL, etc.) that are self-verified by re-running the actual rules against the fixed result before ever calling something "fixed."
  • Also available as: a GitHub Action (comments on PRs), a local CLI (batch-lint files offline, no network call), and a v0.1 VS Code extension. All run the identical rule engine as this page.
